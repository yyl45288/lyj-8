# 社区团购系统架构理解报告

## 1. 项目概述

本项目是一个社区团购电商系统，基于 **Python + FastAPI + SQLAlchemy + SQLite** 技术栈构建。系统覆盖了电商核心业务流程：商品浏览、下单支付、库存管理、订单分拣、配送调度、售后服务等完整链路。

### 1.1 技术架构

```
┌─────────────────────────────────────────────────────────┐
│                        API Layer                        │
│              [app/api/routes.py]  FastAPI Routes        │
└─────────────────────────────────────┬───────────────────┘
                                      │
┌─────────────────────────────────────▼───────────────────┐
│                   Business Service Layer                │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐  │
│  │OrderService │  │Inventory    │  │PricingService   │  │
│  │(order_flow) │  │Service      │  │(Promotion+Coupon│  │
│  └──────┬──────┘  │(stock_lock) │  │  Engine)        │  │
│         │         └──────┬──────┘  └────────┬────────┘  │
│         │                │                    │           │
│  ┌──────▼──────┐  ┌──────▼──────┐  ┌────────▼────────┐  │
│  │Dispatch     │  │Data Models  │  │Database Session │  │
│  │Service      │  │[models.py]  │  │[database.py]    │  │
│  │(route_opt)  │  │SQLAlchemy   │  │SQLite + WAL     │  │
│  └─────────────┘  └─────────────┘  └─────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 2.2 详细调用链路

#### 2.2.1 订单创建流程

```
POST /api/orders
    ↓
routes.create_order()
    ↓
OrderService.create_order()
    ├─► 验证用户、团长、商品存在性
    ├─► PricingService.calculate_price()
    │   ├─► PromotionEngine.apply_promotions()
    │   │   ├─► _apply_group_buy()
    │   │   ├─► _apply_full_reduction()
    │   │   └─► _apply_second_half_price()
    │   └─► CouponEngine.apply_coupon()
    ├─► 创建Order + OrderItem记录
    └─► InventoryService.reserve()  # 锁定库存30分钟
        ├─► _clean_expired_reservations()
        └─► UPDATE Inventory SET reserved_qty += qty
```

#### 2.2.2 订单支付流程

```
POST /api/orders/{id}/pay
    ↓
routes.pay_order()
    ↓
OrderService.pay_order()
    ├─► 验证状态流转 CREATED → PAID
    ├─► 检查库存锁定是否过期
    │   └─► (过期则自动取消订单并释放库存)
    ├─► InventoryService.confirm_reservations()
    │   └─► UPDATE Inventory SET available -= qty, reserved -= qty
    ├─► CouponEngine.mark_coupon_used()
    │   └─► UPDATE UserCoupon SET used=True, used_count++
    └─► 更新订单状态为 PAID
```

#### 2.2.3 截单与分拣流程

```
POST /api/warehouse/cutoff
    ↓
routes.cutoff_orders()
    ↓
OrderService.cutoff_orders()
    ├─► 查询所有 PAID 状态订单
    ├─► 更新状态为 CUTOFF
    └─► _create_sorting_task()
        ├─► 创建 SortingTask (PENDING)
        └─► 创建 SortingItem 明细

POST /api/sorting/tasks/{id}/start
    ↓
OrderService.start_sorting()
    ├─► SortingTask: PENDING → IN_PROGRESS
    └─► Order: CUTOFF → SORTING

POST /api/sorting/tasks/{id}/complete
    ↓
OrderService.complete_sorting()
    ├─► 验证所有SortingItem已完成
    ├─► SortingTask: IN_PROGRESS → COMPLETED
    ├─► Order: SORTING → SORTED
    └─► InventoryService.mark_sorted()  # 标记已分拣数量
```

#### 2.2.4 配送流程

```
POST /api/dispatch/routes
    ↓
routes.create_routes()
    ↓
DispatchService.create_delivery_routes()
    ├─► get_ready_orders()  # SORTED状态 + 分拣完成 + 未派车
    ├─► get_available_vehicles()  # IDLE状态车辆
    ├─► _group_orders_by_district() / _group_orders_by_leader()
    ├─► _split_stops_by_capacity()  # 按车辆容量拆分
    ├─► _optimize_stop_sequence()  # 贪心算法路径优化
    ├─► _create_route()
    │   ├─► 创建 DeliveryRoute
    │   ├─► 创建 DeliveryStop (每个订单一个站点)
    │   ├─► Order: SORTED → DELIVERING
    │   └─► Vehicle: IDLE → LOADING
    └─► 返回路线列表

POST /api/dispatch/routes/{id}/dispatch
    ↓
DispatchService.dispatch_route()
    ├─► 合并同一sequence的重复DeliveryStop
    ├─► DeliveryRoute: PENDING → DISPATCHED
    └─► Vehicle: LOADING → DELIVERING

POST /api/dispatch/stops/{id}/arrive
    ↓
DispatchService.stop_arrived()
    ├─► DeliveryStop.arrived_at = now
    ├─► Order: DELIVERING → DELIVERED
    └─► DeliveryRoute: DISPATCHED → IN_TRANSIT
```

#### 2.2.5 售后流程

```
POST /api/aftersale
    ↓
routes.create_aftersale()
    ↓
OrderService.apply_after_sale()
    ├─► 验证订单状态 (DELIVERED/PICKED_UP/COMPLETED)
    ├─► 检查是否有处理中的售后
    ├─► 创建 AfterSale + AfterSaleItem
    ├─► 计算退款金额
    └─► Order: * → REFUNDING

POST /api/aftersale/{id}/approve
    ↓
OrderService.approve_after_sale()
    ├─► AfterSale: SUBMITTED → APPROVED
    └─► Order: * → REFUNDING (如未设置)

POST /api/aftersale/{id}/refund
    ↓
OrderService.execute_refund()
    ├─► AfterSale: APPROVED → REFUNDED → CLOSED
    ├─► Order: REFUNDING → REFUNDED
    └─► PaymentStatus: REFUNDING → REFUNDED
```

## 3. 订单完整状态流转

### 3.1 订单状态枚举

定义于 [models.py](file:///d:/git/lyj-8/app/models.py#L8-L20)：

| 状态 | 说明 |
|------|------|
| `CREATED` | 已创建（待支付） |
| `PAID` | 已支付 |
| `CUTOFF` | 已截单 |
| `SORTING` | 分拣中 |
| `SORTED` | 分拣完成 |
| `DELIVERING` | 配送中 |
| `DELIVERED` | 团长签收 |
| `PICKED_UP` | 用户已提货 |
| `COMPLETED` | 已完成 |
| `CANCELLED` | 已取消 |
| `REFUNDING` | 退款中 |
| `REFUNDED` | 已退款 |

### 3.2 状态流转规则

定义于 [order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py#L20-L33)：

```python
STATE_TRANSITIONS = {
    CREATED:     [PAID, CANCELLED],
    PAID:        [CUTOFF, CANCELLED, REFUNDING],
    CUTOFF:      [SORTING, REFUNDING],
    SORTING:     [SORTED],
    SORTED:      [DELIVERING],
    DELIVERING:  [DELIVERED],
    DELIVERED:   [PICKED_UP],
    PICKED_UP:   [COMPLETED],
    REFUNDING:   [REFUNDED, PAID],  # PAID是回滚分支
    REFUNDED:    [],
    CANCELLED:   [],
    COMPLETED:   [],
}
```

### 3.4 异常状态回流

1. **支付时库存锁定过期**：`CREATED` → 自动 `CANCELLED`，释放库存
2. **售后拒绝**：`REFUNDING` → `COMPLETED` / `DELIVERED`，恢复支付状态
3. **取消已支付订单**：`PAID` → `CANCELLED`，库存回库，优惠券回滚
4. **取消未支付订单**：`CREATED` → `CANCELLED`，释放库存锁定

## 4. 关键风险点分析

### 4.1 库存账实不符风险

**风险场景**：系统库存记录与实际仓库库存不一致

**高风险代码位置**：

1. **[inventory.py](file:///d:/git/lyj-8/app/modules/inventory.py) `reserve()` 方法 (L127-L176)**
   - 风险点：`get_or_create_inventory()` 调用时 `lock=False`，未使用行锁
   - 并发场景下可能导致超卖（虽有乐观更新 `WHERE available - reserved >= qty`，但无数据库行锁保护）
   - 虽然测试 `test_concurrent_purchase_no_supersell` 验证了无超卖，但高并发下仍有理论风险

2. **[inventory.py](file:///d:/git/lyj-8/app/modules/inventory.py) `confirm_reservations()` 方法 (L178-L228)**
   - 风险点：`actual_decrease = min(reservation.qty, inventory.available_qty, inventory.reserved_qty)`
   - 如果 `available` 或 `reserved` 不足，会静默减少扣减数量而不报错
   - 可能导致"部分确认"，库存账目混乱

3. **[inventory.py](file:///d:/git/lyj-8/app/modules/inventory.py) `_merge_duplicate_inventories()` 方法 (L80-L100)**
   - 风险点：当出现重复库存记录时才会合并，说明存在重复创建的可能性
   - 虽然 `UniqueConstraint` 存在，但 `get_or_create_inventory` 的并发创建仍可能导致暂时重复

4. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `cancel_order()` 方法 (L185-L220)**
   - 风险点：`PAID` 状态取消时调用 `return_stock()` 增加 `available_qty`
   - 但未同步扣减 `sorted_qty`（如果已分拣）
   - 如果订单已分拣后取消，会导致 `sorted_qty` 虚高

5. **[inventory.py](file:///d:/git/lyj-8/app/modules/inventory.py) `mark_sorted()` 方法 (L400-L415)**
   - 风险点：`new_sorted = min(inventory.sorted_qty + qty, inventory.available_qty + inventory.sorted_qty)`
   - 允许 `sorted_qty` 超过 `available_qty`，与实际业务逻辑不符
   - 正常逻辑下已确认的库存（`available` 已扣减）才能分拣

**根本原因**：
- 库存操作分散在多个方法中，缺乏统一的事务边界
- 部分场景使用 `update ... where` 乐观锁，部分场景直接读取后更新
- `sorted_qty` 字段的业务含义不清晰（是已分拣的数量，还是分拣中的数量？）

### 4.2 优惠券状态错误风险

**风险场景**：优惠券被标记为已使用，但订单最终取消/退款，优惠券未正确回滚

**高风险代码位置**：

1. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `pay_order()` 方法 (L137-L183)**
   - 风险点：`self.coupon_engine.mark_coupon_used()` 在 `confirm_reservations()` 之后执行
   - 如果库存确认成功但优惠券标记失败（异常），事务回滚但无重试机制

2. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `cancel_order()` 方法 (L199-L200)**
   - 风险点：仅在 `status == PAID` 时才回滚优惠券
   - 如果订单已进入 `CUTOFF` / `SORTING` 等后续状态后取消，优惠券不会回滚
   - 但 `STATE_TRANSITIONS` 中 `PAID` 之后的状态不允许直接 `CANCELLED`，理论上不会发生

3. **[pricing.py](file:///d:/git/lyj-8/app/modules/pricing.py) `unmark_coupon_used()` 方法 (L327-L344)**
   - 风险点：未检查 `used_order_id` 是否匹配当前订单
   - 如果同一张优惠券被误用（理论上不会发生），回滚时可能错误释放

4. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `apply_after_sale()` 方法 (L410-L480)**
   - 风险点：售后退款时未处理优惠券回滚
   - 订单退款后，已使用的优惠券应根据业务规则决定是否返还

5. **[pricing.py](file:///d:/git/lyj-8/app/modules/pricing.py) `mark_coupon_used()` 方法 (L307-L325)**
   - 风险点：查询 `UserCoupon` 时使用 `used == False` 过滤，但更新时没有 `WHERE used == False` 条件
   - 并发场景下可能出现"重复标记"（两个支付请求同时进入，都查到未使用，都标记为已使用）

**根本原因**：
- 优惠券状态变更缺乏数据库层面的乐观锁/悲观锁保护
- 售后流程未完整考虑优惠券的生命周期管理
- 状态变更缺少幂等性校验

### 4.3 重复分拣风险

**风险场景**：同一订单被多次创建分拣任务，或同一分拣任务被多次执行

**高风险代码位置**：

1. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `cutoff_orders()` 方法 (L222-L245)**
   - 风险点：多次调用 `cutoff_orders()` 会重复创建分拣任务
   - 没有检查该订单是否已存在分拣任务
   - 虽然状态校验 `_validate_transition` 会阻止重复更新订单状态，但 `SortingTask` 会重复创建

2. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `_create_sorting_task()` 方法 (L247-L268)**
   - 风险点：直接创建 `SortingTask`，无任何重复性检查
   - 一个订单可能对应多个分拣任务

3. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `start_sorting()` 方法 (L270-L292)**
   - 风险点：检查 `task.status != SortingStatus.PENDING` 但未加锁
   - 并发请求可能同时通过检查，导致状态混乱

4. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `sort_item()` 方法 (L294-L311)**
   - 风险点：`item.sorted_qty = min(item.sorted_qty + qty, item.required_qty)`
   - 没有幂等性保护，同一批次可能被重复扫码分拣
   - 缺少分拣操作日志，无法追溯

5. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `complete_sorting()` 方法 (L313-L341)**
   - 风险点：循环调用 `mark_sorted()` 更新库存，无原子性保证
   - 部分更新成功部分失败会导致数据不一致

**根本原因**：
- 缺少 `order_id + warehouse_date` 的唯一约束来防止重复分拣任务
- 分拣操作缺乏幂等性设计（如使用分拣批次号去重）
- 状态变更没有数据库层面的乐观锁（`WHERE status = old_status`）

---

### 4.4 重复派车风险

**风险场景**：同一订单被分配到多条配送路线，或同一条路线被多次发车

**高风险代码位置**：

1. **[dispatch.py](file:///d:/git/lyj-8/app/modules/dispatch.py) `get_ready_orders()` 方法 (L62-L80)**
   - 风险点：通过 `get_dispatched_orders()` 查询已派车订单，使用内存过滤
   - 并发调用 `create_delivery_routes()` 可能导致同一订单被多次分配
   - 缺少数据库层面的"已派车"标记或唯一约束

2. **[dispatch.py](file:///d:/git/lyj-8/app/modules/dispatch.py) `create_delivery_routes()` 方法 (L144-L216)**
   - 风险点：先查询后分配，无事务锁保护
   - 高并发下可能出现"幻读"，同一订单被多次加入路线

3. **[dispatch.py](file:///d:/git/lyj-8/app/modules/dispatch.py) `_create_route()` 方法 (L237-L278)**
   - 风险点：直接将 `order.status` 更新为 `DELIVERING`
   - 如果创建路线过程中出现异常，部分订单状态可能已更新
   - 没有回滚机制

4. **[dispatch.py](file:///d:/git/lyj-8/app/modules/dispatch.py) `dispatch_route()` 方法 (L280-L317)**
   - 风险点：检查 `route.status != DeliveryStatus.PENDING` 但无乐观锁
   - 并发发车请求可能重复更新车辆状态
   - `Vehicle.status` 更新时缺少 `WHERE status = LOADING` 条件

5. **[dispatch.py](file:///d:/git/lyj-8/app/modules/dispatch.py) `stop_arrived()` 方法 (L319-L349)**
   - 风险点：循环更新同 sequence 的其他站点状态
   - 如果有一个站点更新失败，其他站点可能已更新，导致不一致
   - 同时更新 `Order.status`，缺少状态流转校验

**根本原因**：
- 派车逻辑缺少分布式锁或数据库行锁保护
- 订单"已派车"状态依赖 `DeliveryStop` 存在性，而非订单自身状态字段
- 批量更新操作缺少原子性保证

### 4.5 重复退款风险

**风险场景**：同一售后申请被多次执行退款，或同一订单被多次售后

**高风险代码位置**：

1. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `apply_after_sale()` 方法 (L410-L480)**
   - 风险点：检查已有处理中的售后时，状态过滤为 `SUBMITTED/APPROVED/REFUNDING`
   - 如果前一次售后已 `REFUNDED` 但未 `CLOSED`，可能允许重复申请
   - 但代码中 `execute_refund()` 会设置为 `CLOSED`，理论上不会发生

2. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `execute_refund()` 方法 (L526-L548)**
   - 风险点：
     ```python
     after_sale.status = AfterSaleStatus.REFUNDED
     after_sale.refunded_at = datetime.utcnow()
     # ... 更新订单状态 ...
     after_sale.status = AfterSaleStatus.CLOSED  # 连续两次更新状态
     after_sale.closed_at = datetime.utcnow()
     ```
   - 状态变更缺少 `WHERE status = APPROVED` 条件
   - 并发请求可能导致重复退款

3. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `cancel_order()` 方法 (L213-L217)**
   - 风险点：取消已支付订单时自动标记为退款完成
   - 没有调用实际的退款接口，仅更新本地状态
   - 与 `execute_refund()` 逻辑重复，可能导致双重退款

4. **[order_service.py](file:///d:/git/lyj-8/app/modules/order_service.py) `execute_refund()` 方法 (L536-L542)**
   - 风险点：更新订单状态时，不管原状态是什么直接设为 `REFUNDED`
   - 如果订单已完成其他流程，可能覆盖重要状态
   - 缺少库存回库逻辑（与 `cancel_order` 不同）

5. **[routes.py](file:///d:/git/lyj-8/app/api/routes.py) `execute_refund()` 接口 (L492-L501)**
   - 风险点：API层缺少幂等性设计（如使用退款流水号）
   - 重复调用可能导致多次退款

**根本原因**：
- 退款操作缺少幂等键（Idempotency Key）保护
- 状态变更没有使用 `UPDATE ... WHERE` 乐观锁
- `cancel_order` 和 `execute_refund` 都涉及退款，逻辑分散，容易不一致
- 缺少实际的支付网关退款调用，仅更新本地状态

## 6. 总结与建议

### 6.1 架构优点

1. **分层清晰**：API层 → 服务层 → 数据层，职责分离明确
2. **状态机设计**：订单状态流转有明确的规则定义，便于维护
3. **库存锁定机制**：使用 `available - reserved` 模式，支持30分钟支付超时
4. **并发防护**：库存操作使用 `UPDATE ... WHERE` 乐观锁，多商品按ID排序防止死锁
5. **数据库配置**：SQLite启用WAL模式，支持更高并发

### 6.2 关键改进建议

| 优先级 | 问题 | 改进方案 |
|--------|------|---------|
| 🔴 高 | 库存并发安全 | 所有库存变更操作添加 `with_for_update()` 行锁 |
| 🔴 高 | 优惠券重复使用 | `mark_coupon_used` 添加 `WHERE used = False` 乐观锁 |
| 🔴 高 | 重复分拣任务 | `SortingTask` 添加 `order_id + warehouse_date` 唯一约束 |
| 🔴 高 | 重复派车 | `DeliveryStop` 添加 `order_id` 唯一约束，或订单增加 `dispatched` 标记 |
| 🔴 高 | 重复退款 | `execute_refund` 添加 `WHERE status = APPROVED` 乐观锁 |
| 🟠 中 | 状态变更无锁 | 所有状态更新使用 `UPDATE ... WHERE status = old_status` |
| 🟠 中 | 幂等性缺失 | 关键接口（支付、退款、发车）添加幂等键支持 |
| 🟠 中 | 测试覆盖不足 | 按第5章建议补充高风险场景测试 |
| 🟡 低 | 代码重复 | `cancel_order` 和 `execute_refund` 的退款逻辑统一 |
| 🟡 低 | 库存字段歧义 | 明确 `sorted_qty` 的业务含义，补充文档 |

### 6.3 监控建议

建议增加以下业务监控指标：
1. 库存账实差异率（定期盘点对比）
2. 优惠券状态不一致数量（已使用但无对应已支付订单）
3. 重复分拣任务数量
4. 重复派车订单数量
5. 退款重试/重复调用次数
6. 各状态订单数量分布
